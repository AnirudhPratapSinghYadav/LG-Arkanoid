# LG Arkanoid

A multiplayer panoramic brick breaker that runs across all five screens of a Liquid Galaxy rig as one continuous 9600 pixel wide arcade wall.

**Repository:** [github.com/AnirudhPratapSinghYadav/LG-Arkanoid](https://github.com/AnirudhPratapSinghYadav/LG-Arkanoid)

## What is LG Arkanoid

LG Arkanoid brings the classic brick breaker arcade experience to the Liquid Galaxy panoramic display wall. The entire five screen rig becomes one continuous game canvas spanning 9600 virtual pixels. Bricks stretch across all screens. The ball travels freely from Screen 1 through Screen 5, crossing physical bezels without stuttering or disappearing.

Up to three players each control a paddle from an Android phone using the Flutter controller app. No game graphics render on the phone. Everything plays out on the rig. When the ball crosses a screen boundary, both the departing and arriving screens update in the same frame at 60 frames per second.

Screen 5 also displays AI generated retro arcade commentary powered by Gemini, giving the experience a live announcer feel across the panoramic wall.

## How it Works

LG Arkanoid is built on four layers that work together.

**Layer one: Flutter controller app.** Each player runs the LG Arkanoid app on their Android phone. The app provides a horizontal drag strip for paddle control, power up buttons, and a live score and lives display. It connects to the game server over Socket.io with websocket transport. The phone sends input and receives score updates and commentary text.

**Layer two: Node.js game server.** The server runs on the LG master node and holds the only authoritative copy of the game world. This includes ball positions and velocities, the full brick grid, paddle positions, scores, lives, and session tokens. The server runs a 16 millisecond physics loop and broadcasts game state to all clients at 60 frames per second.

**Layer three: Phaser screen clients.** Five Chromium browser instances each run a Phaser 3 HTML5 Canvas client. Each instance renders only the slice of the virtual canvas assigned to its screen. Clients hold zero local physics state. They receive game state from the server and draw bricks, balls, and paddles every frame.

**Layer four: Gemini commentary.** When significant game events occur such as level clears, life losses, multi ball activation, score milestones, and victory, the server calls the Gemini API to generate 15 words of retro arcade announcer commentary. The text appears on Screen 5 and on connected phone status screens. If the API is unavailable, a local fallback pool of 20 commentary lines is used instead.

## Key Engineering Challenge

The ball boundary crossing problem is the central technical challenge. When the ball reaches the right edge of Screen 2, it must appear at the left edge of Screen 3 at the exact matching vertical position within a single 16 millisecond frame.

The Node.js server handles this centrally. It detects when the ball's next position would cross a screen boundary, computes the entry coordinates on the adjacent screen, and emits boundary_exit to the departing screen and boundary_enter to the arriving screen synchronously in the same execution. Both screens acknowledge within 16 milliseconds. If either acknowledgment is missing, the server retries once.

This ensures the ball never vanishes at a bezel seam and the panoramic rig feels like one uninterrupted arcade cabinet.

## Technology Stack

Game rendering uses Phaser 3 and HTML5 Canvas with WebGL and canvas fallback. The authoritative game server runs on Node.js with Express and Socket.io. The mobile controller is built with Flutter and Dart using socket_io_client for real time communication. Commentary is generated by Gemini 2.0 Flash via the Google Generative Language API using redeemed GCP credit. Shared preferences store the last server address on the phone for quick reconnection.

## Repository Structure

```
LG-Arkanoid/
├── server/              Node.js game server
├── web client/          Phaser 3 screen clients
├── flutter app/         Flutter phone controller app
├── docs/                Protocol and architecture documentation
└── README.md
```

**server/** contains the full game server including physics, security validation, boundary handoff, and Gemini commentary.

**web client/** contains a single HTML file loaded by each Chromium instance on the rig with a screenId URL parameter.

**flutter app/** contains the Android controller with connect, controller, and status screens.

**docs/** contains the protocol reference and architecture documentation for mentors and contributors.

## Getting Started

**1. Start the game server**

```bash
cd server
npm install
node index.js
```

The server listens on port 8080 and logs `LG Arkanoid game server running on port 8080`.

Optionally set your Gemini API key for live commentary:

```bash
export GEMINI_API_KEY=your_key_here
node index.js
```

**2. Open screen clients on the rig**

Open five Chromium browser windows, one per physical screen:

```
http://<master-node-ip>:8080/?screenId=1
http://<master-node-ip>:8080/?screenId=2
http://<master-node-ip>:8080/?screenId=3
http://<master-node-ip>:8080/?screenId=4
http://<master-node-ip>:8080/?screenId=5
```

Press S on any screen to start a new game. The 6 digit session token appears on all screens for 10 seconds. Screen 5 is the best place for players to read it.

**3. Connect phone controllers**

```bash
cd "flutter app"
flutter pub get
flutter run
```

Enter the master node IP, port 8080, and the 6 digit session token shown on Screen 5. Drag horizontally to move your paddle. Use Power Up and Fire Ball buttons during play.

**4. Health check**

```
GET http://localhost:8080/health
```

Returns server status and connected player count.
